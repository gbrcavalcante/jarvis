# Implementation Plan: Pluggable Agent Backends

**Branch**: `005-pluggable-agent-backends` | **Date**: 2026-06-24 | **Spec**: [spec.md](spec.md)

## Summary

Add a backend selection layer above the existing provider router. Users register and switch between agent backends (Built-in Router, OpenClaw, Hermes Agent, LangGraph) entirely through the Settings Panel. A single `ExternalHttpAgent` adapter communicates with external backends via the OpenAI-compatible `/v1/chat/completions` endpoint вЂ” no code changes required to support new backends. All credentials stored in OS keychain.

## Technical Context

**Language/Version**: Python 3.11+

**Primary Dependencies**: FastAPI (existing), SQLAlchemy + aiosqlite (existing), httpx (existing), PyQt6 (existing), pybreaker (new вЂ” circuit breaker for health polling)

**Storage**: SQLite via SQLAlchemy вЂ” two new tables: `agent_backends`, `backend_dispatch_events`. Credentials in OS keychain only.

**Testing**: pytest + pytest-asyncio (existing). New tests under `tests/unit/agents/`, `tests/unit/api/`, `tests/unit/ui/`

**Target Platform**: Linux desktop + Windows desktop (existing)

**Performance Goals**: Backend health check polling adds < 1 ms overhead per request (check runs async, cached result used). Backend switch takes effect within one request cycle.

**Constraints**: Zero changes to existing `BaseAgent` interface or `Router`. Built-in Router always available as fallback. No restart required for any backend management action.

**Scale/Scope**: Up to ~10 registered backends per installation.

## Implementation Phases

### Phase 1 вЂ” Data layer
- `AgentBackend` and `BackendDispatchEvent` SQLAlchemy models
- DB migration: create tables, seed Built-in Router row
- `BackendStore` async CRUD helper

### Phase 2 вЂ” ExternalHttpAgent adapter
- `ExternalHttpAgent(BaseAgent)`: `complete()`, `stream()`, `is_available()`, `cancel()`
- OpenAI-compatible JSON serialization
- LangGraph normalization (`/agent/invoke` в†’ standard shape)
- `pybreaker` circuit breaker on `is_available()`

### Phase 3 вЂ” REST API routes
- `GET /backends`, `POST /backends`, `PATCH /backends/{id}`, `DELETE /backends/{id}`
- `POST /backends/active`, `POST /backends/{id}/test`
- Credentials routed to OS keychain, never stored in DB

### Phase 4 вЂ” Router integration
- Router reads active `AgentBackend` at dispatch time
- If active backend is external: instantiates `ExternalHttpAgent`, dispatches
- If active backend is Built-in Router: existing logic unchanged
- On failure: falls back to Built-in Router, writes `BackendDispatchEvent`, logs

### Phase 5 вЂ” Health monitoring
- Background `asyncio` task polls every 10 s
- Updates `AgentBackend.health_status`, `error_count`, `last_seen_at`
- `pybreaker` opens after 3 consecutive failures, resets after 30 s

### Phase 6 вЂ” Settings Panel UI
- New "Agents" tab in `src/ui/sections/agents.py`
- Backend list with health indicators (green/yellow/red dot)
- Add / Edit / Remove / Test Connection / Set Active buttons
- Per-backend form with name, URL, model, API key fields

### Phase 7 вЂ” Coverage & integration
- End-to-end test: register в†’ test в†’ activate в†’ dispatch в†’ verify audit log
- Coverage check в‰Ґ 80%
